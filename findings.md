## 1. Game Rules

Plinko is a variable-variance game where a ball drops through a triangular field of pins, bouncing left or right at each row before landing in a numbered slot at the bottom. Each slot carries a multiplier that determines the payout.

**Player choices:** bet amount, risk level (Low / Medium / High), number of rows (8–16).

**Outcome:** the ball lands in a slot numbered 0 to n (where n = number of rows). The slot determines the payout multiplier.

**Configurations:** 27 total (3 risk levels × 9 row counts).

| Parameter | Value |
|-----------|-------|
| Row configurations | 8, 9, 10, 11, 12, 13, 14, 15, 16 |
| Risk levels | Low, Medium, High |
| House edge (bracket 0) | 0.1% (99.9% RTP) |
| Minimum bet tested | $0.01 USDT |
| RNG algorithm | HMAC-SHA256 |
| Seed format | Server: 64-char hex (32 bytes); Client: alphanumeric string |
| Epoch size | 50 bets per seed rotation |

**Slot distribution** follows binomial B(n, 0.5): P(slot = k) = C(n, k) / 2^n

| Rows | Total Slots | P(center) | P(edge) |
|------|-------------|-----------|---------|
| 8 | 9 (0–8) | 27.34% | 0.39% |
| 12 | 13 (0–12) | 22.56% | 0.024% |
| 16 | 17 (0–16) | 19.64% | 0.0015% |

**Multiplier structure** is symmetric — slot k and slot (rows − k) always carry the same multiplier. All three risk levels target the same RTP (~99.9%) but differ in variance:

| Risk | Center Multiplier (16-row) | Edge Multiplier (16-row) |
|------|---------------------------|--------------------------|
| Low | 0.50× | 16.15× |
| Medium | 0.30× | 111.01× |
| High | 0.20× | 1009.33110594× |

**Progressive edge structure:** `plinkoConfig.json` contains 191 bet-size brackets per configuration. Bracket 0 (house edge 0.001 = 0.1%) covers bets up to ~$335.95 for the 16r/high configuration (the most restrictive; other configs have higher ceilings). Higher brackets carry progressively higher house edges, reaching 0.02 (2.0%) at the highest bracket. All bets in this audit ($0.01 and $10) fall within bracket 0.

## 2. Algorithm

The outcome is computed using HMAC-SHA256 with one HMAC call per row. Each row produces an independent left/right bounce.

### Inputs

| Input | Source | Purpose |
|-------|--------|---------|
| Server Seed | Generated by Duel (64-char hex) | Primary entropy — hidden until rotation |
| Client Seed | Set by player (alphanumeric) | Player-contributed randomness |
| Nonce | Auto-incrementing integer (0–49) | Unique output per bet within epoch |
| Rows | Selected by player (8–16) | Number of bounces |

### Steps

```
position = 0
for cursor = 0 to (rows - 1):
    message = "{clientSeed}:{nonce}:{cursor}"
    hash = HMAC-SHA256(hexToBytes(serverSeed), UTF8(message))
    value = readUInt32BE(hash, offset=0)
    position += (value % 2)
final_slot = position   // 0 to rows
payout = bet × multiplier_table[risk][rows][final_slot]
```

**Critical detail:** The server seed is hex-decoded to 32 raw bytes before use as the HMAC key. Using the 64-char hex string as UTF-8 (which produces 64 bytes) gives completely different results.

### Bias Analysis

`value % 2` on a uint32 has mathematically zero modulo bias — 2^32 is exactly divisible by 2, so exactly half the values produce 0 and half produce 1. No rejection sampling needed.

Each row uses a unique cursor in the HMAC message, so per-row outcomes are statistically independent. HMAC-SHA256 is a PRF — knowing one row's output gives zero information about other rows.

## 5. RTP Analysis

Formula: RTP = Σ P(slot\_k) × multiplier(slot\_k) where P(slot\_k) = C(n, k) / 2^n

Uses `scaling_edge[0].multipliers` (API-precision) for all calculations.

## 6. Statistical Distribution Tests

### Chi-Squared Goodness-of-Fit (Simulation)

Tested all 27 configurations. For each, ran 1,000,000 simulated rounds and compared observed slot frequencies against independent binomial B(n, 0.5) expected counts. Alpha = 0.01 is the per-test threshold across 27 independent configurations. The family-wise error rate (FWER) at this threshold is 1−0.99²⁷ ≈ 23.8%. With Bonferroni correction, the per-test threshold would be α/27 ≈ 0.00037. One config (8r/high, p=0.0059) falls below the uncorrected α=0.01 — a single-run false positive within the expected FWER rate for 27 independent tests. All 27 configs pass the Bonferroni-corrected threshold (minimum p=0.0059 > 0.00037). The choice of correction method does not change the conclusion: 0/27 reject the null hypothesis at the Bonferroni threshold.

### Chi-Squared on Live Bets

Primary check using actual captured bets (alpha = 0.01, slots with expected count < 5 pooled). **All 27 configurations pass.** Source: `outputs/chi-squared-results.json`.

**Notable result — 16r/low (n=200):** χ² = 20.45, df = 10, p = 0.0252. This is the weakest result across all 27 live configs. The decision threshold for this audit is α=0.01; this result passes. The p = 0.0252 value is noted for transparency because it falls below the conventional secondary threshold of α=0.05. Under H₀ (all configs truly binomial), the probability of observing at least one p-value below 0.05 across 27 independent tests is 1 − 0.95²⁷ ≈ 75% — so a result in this range is statistically expected and does not indicate manipulation. The 16r/low simulation (1,000,000 rounds) produces χ² = 10.2, df = 16, p = 0.857 in the current simulation run, consistent with H₀.

## 7. Anomalies, Observations & Red Flags

### Anomalies

**effective\_edge field**

The API returns `effective_edge` on each bet at two values: 0.1 (6,983 bets, 91.9%) and 0 (617 bets, 8.1%). The multiplier tables are identical across both values — same (risk, rows, slot) always returns the same payout multiplier regardless of `effective_edge`. The zero-edge mechanism is a transaction-level rebate applied outside game logic. Game outcomes, RNG, slot determination, and payout calculation are unaffected.

**Capture-retry nonce pattern**

5 of 152 epochs show a capture-retry artifact. In each, the server processed 51 bets (nonces 0–50); the capture script recorded 50 of them, missing one nonce in the 0–49 range (the specific missed nonce per epoch is logged in `outputs/verification-results.json` step 4). Nonce 50 from the server's response was recorded instead. Slot recomputation passes for all 50 captured bets in each affected epoch. This is a collection artifact, not a server anomaly. Affected epoch hashes: `645b424d5b995f54`, `91b453380f29025f`, `358dd868189fb265`, `e1703ae848174c7a`, `d607aad19bd588aa`.

**drand fields**

`drand_round` and `drand_randomness` are not required for Plinko slot computation. RNG depends solely on (serverSeed, clientSeed, nonce, cursor). Confirmed by slot recomputation: all 7,600 slots independently reproduced without referencing drand inputs. 0 mismatches.

**Progressive house edge**

`plinkoConfig.json` defines 191 bet-size brackets per configuration. House edge scales from 0.001 (0.1%) at bracket 0 (bets up to ~$335.95 for 16r/high, the most restrictive config) to 0.02 (2.0%) at the highest bracket. All test bets ($0.01 and $10) fall in bracket 0. The multiplier table used by the server matches `scaling_edge[0].multipliers` for all 7,600 bets.

**Plan deviation — Phase A config distribution (disclosed)**

The audit plan specified 200 bets per configuration (27 × 200 = 5,400 total) with equal coverage. This target was not fully met: 10 of 27 configs have fewer than 200 bets due to a localStorage quota event at approximately bet 4,934, which caused the capture session to restart from a new randomly selected starting configuration. Minimum coverage: 116 bets (11r/high). The total Phase A count (5,400) is correct.

Impact assessment: All 5,400 bets are individually verified via slot recomputation (EC-7). The minimum of 116 bets exceeds the chi-squared validity threshold of 100 (expected counts ≥ 5 after pooling). The imbalance does not introduce bias — the affected configs are not systematically enriched or depleted from one risk level. The per-config RTP z-test uses the correct n for each config's sample. No chi-squared or RTP test conclusion changes at any of the affected configs.

**Console capture method**

Data was collected via a console-injected JavaScript script (`capture/plinko-capture.js`), not a Tampermonkey userscript. The capture script intercepts `fetch` responses directly. No functional difference in captured data.

### Red Flags

None. No suspicious nonce behaviour, no unexpected seed changes mid-epoch, no multiplier inconsistencies, no anomalous timing patterns in any of the 7,600 bets or 152 epochs.

Post-collection, all 3 plaintext server seeds were retrieved via `GET /api/v2/user/transactions/{txId}`. Hash verification: 3/3 pass. Slot recomputation: 150/150 bets match. The dataset is now 100% verified — 7,600/7,600 bets with zero slot mismatches.

**2. 50-bet epoch size**

The checklist specifies 100-bet seed rotation intervals. Duel.com's server enforces 50-bet epochs — the platform automatically rotates seeds after exactly 50 bets regardless of client instruction. This is a platform constraint, not a collection protocol deviation.

**3. 200 bets at $10 (Phase C)**

Phase C was a code-path equivalence check, not a standalone RTP study. Its purpose was to confirm that $10 bets use the same HMAC-SHA256 path and the same `scaling_edge[0].multipliers` table as $0.01 bets. For this goal, 200 bets is sufficient: all 200 recomputed correctly from revealed seeds, and all 200 multipliers matched the reference table within 1e-8. Equivalence is proven deterministically. No distributional test is applied — at n=200, statistical comparisons have near-zero power and add nothing over the deterministic proof. The conclusion (same code path, no hidden bet-size branch) rests on 200/200 exact matches.

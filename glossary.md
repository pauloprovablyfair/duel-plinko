# Glossary

**Audit:** Duel.com Plinko\
**Version:** 1.0\
**Date:** February 2026\
**Auditor:** [ProvablyFair.org](https://www.provablyfair.org)

---

**Provably Fair** — A cryptographic scheme in which the game outcome can be independently verified by the player after the fact using seeds that were committed to before the game. The player cannot alter the server seed; the server cannot alter it after committing; neither party can manipulate the result without detection.

**HMAC-SHA256** — Hash-based Message Authentication Code using SHA-256. Produces a 256-bit (32-byte) deterministic output given a key and message. Used here to generate one bit of Plinko path per row: one HMAC call per cursor position, first 8 hex characters extracted, parity of the resulting integer determines ball direction.

**Server Seed** — A random 64-character hexadecimal string generated by the server. Kept secret during betting. Revealed after epoch rotation. Used as the HMAC key decoded to raw bytes: `Buffer.from(serverSeed, 'hex')`.

**Server Seed Hash** — `SHA-256(hexDecode(serverSeed))`. Published before bets are placed. Players can verify the revealed server seed matches this commitment using the browser snippet in the Reproducibility Guide. A mismatch would prove tampering.

**Client Seed** — A string set by the player (or randomly generated by the platform). Included in the HMAC message as the first field. Changing the client seed changes every slot outcome under the same server seed. Rotated together with the server seed at epoch boundaries.

**Nonce** — An incrementing integer starting at 0. Identifies the bet's position within an epoch. Normal range: 0–49 (50 bets per epoch). Included in the HMAC message as `clientSeed:nonce:cursor`. Five bets in the dataset carry nonce 50 due to a capture-retry pattern; slot recomputation passes for all 5 — they are flagged as collection artifacts, not excluded from verification.

**Cursor** — Row index, zero-based, ranging from 0 to rows−1. One HMAC call is made per cursor per bet. The first 8 hex characters of each HMAC output determine the ball's direction at that row.

**Epoch** — A sequence of exactly 50 bets under one (serverSeed, clientSeed) pair, nonces 0–49. At epoch end the server rotates both seeds and reveals the previous server seed. The dataset covers 152 epochs across all three collection phases.

**Commit-Reveal** — The cryptographic scheme underlying provably fair: the server commits to `serverSeedHashed` before any bets are placed, then reveals `serverSeed` after the epoch closes. The auditor verifies `SHA-256(hexDecode(serverSeed)) === serverSeedHashed` for every epoch with a revealed seed. All 152 seed epochs in this dataset passed.

**Slot** — The ball's final landing position, an integer ranging from 0 (leftmost) to N (rightmost) for an N-row board. Computed as the sum of N independent Bernoulli trials (odd HMAC output = right = 1, even = left = 0). Follows binomial distribution B(N, 0.5) for a fair RNG.

**RTP (Return to Player)** — `sum(win_amount) / sum(bet_amount)` over a set of bets. Empirical RTP across the 7,600-bet dataset is consistent with the theoretical value. Theoretical RTP for all 27 Plinko configurations at bracket 0 is ≈ 99.9% (0.1% house edge), confirmed by 27M-round Monte Carlo simulation (average simulated RTP 99.890%; 27/27 chi-squared pass at α=0.01).

**Scaling Edge** — Duel.com's progressive multiplier bracket system. `plinkoConfig.json` defines 191 bet-size brackets per configuration. House edge scales from 0.001 (bracket 0) to 0.02 (highest bracket). Bracket 0 ceiling for the 16-row high-risk configuration is $335.946312864870400000. All bets in the dataset fall in bracket 0 (bet sizes $0.01 and $10.00).

**drand** — A distributed randomness beacon. `drand_round` and `drand_randomness` are not required for Plinko slot computation. drand plays no role in Plinko's RNG. Confirmed by 7,600/7,600 slot reproductions using only serverSeed, clientSeed, nonce, and cursor — all slots matched without referencing drand inputs. (drand is used in other Duel.com games.)

**Chi-squared test** — Statistical test comparing observed slot frequencies to expected frequencies under a fair binomial distribution. Used in EC-9 (slot symmetry). Slots with expected count < 5 are pooled with adjacent slots before the test to satisfy the minimum expected frequency assumption. Result across all 27 configurations: 0 failures.

**Wilson-Hilferty approximation** — Method for approximating chi-squared p-values using a normal distribution transformation: `z = ((χ²/df)^(1/3) - (1 - 2/(9·df))) / sqrt(2/(9·df))`. Used in `src/stats.ts` to compute p-values without a scipy or statistics library dependency.

**Lag-1 Autocorrelation** — Pearson correlation between consecutive slot outcomes in a sequence. Expected value ≈ 0 for independent RNG. Observed in the Phase B dataset (2,000 bets, 16r/high): r = −0.008860896852259473. Threshold at N = 2,000: ±0.06708203932499368. Result: within bounds, no serial correlation detected.

**Wald-Wolfowitz Runs Test** — Non-parametric test for serial independence. The slot sequence is converted to binary (win/loss based on a threshold), then runs (consecutive sequences of the same outcome) are counted and compared to the expected count under independence. Observed: 640 runs, expected 646.975, z = −0.483, p = 0.629. Result: no serial dependence detected.

**KS Statistic (Kolmogorov-Smirnov)** — Measures the maximum absolute difference between two empirical cumulative distribution functions. The `ksStat()` function is implemented in `src/stats.ts` but is not used as evidence in this audit. Phase C equivalence is established deterministically (200/200 exact slot recomputation + multiplier table identity), which provides stronger proof than any distributional comparison at n=200.

**Phase A** — 5,400 bets at $0.01 across all 27 configurations (4 epochs × 50 bets each per config, with random row/risk assignment). Purpose: full configuration coverage, confirming every combination of rows and risk level is operational and produces correct payouts.

**Phase B** — 2,000 bets at $0.01 fixed to the 16-row high-risk configuration. Purpose: deep high-variance sampling with sufficient sample size for lag-1 autocorrelation, Wald-Wolfowitz runs test, and slot chi-squared testing.

**Phase C** — 200 bets at $10.00 fixed to the 16-row high-risk configuration. Purpose: code-path equivalence check — verify the RNG produces amount-independent outcomes by comparing slot distributions against Phase B using the KS statistic.

**EC (Edge Case)** — A numbered audit check as defined in `PLAN.md`. 31 EC definitions covering RNG correctness (EC-1 through EC-7), statistical independence (EC-9 through EC-14), exploit resistance and code-path equivalence (EC-15 through EC-25), and dataset integrity (EC-26 through EC-32).

**Hard Fail** — Any EC result that immediately invalidates the provably fair claim, regardless of other results. Six hard-fail criteria: EC-1 (seed hash integrity), EC-4 (nonce monotonic), EC-5 (client seed stable), EC-7 (slot recomputation), EC-26 (hash stable within epoch), EC-27 (client seed actually used). None triggered in this audit.

**Capture-Retry Pattern** — Observed in 5 epochs: the server processed a bet and incremented the nonce, but the client capture script missed the HTTP response and retried the request, receiving nonce 50 in the retry response. The skipped nonce is not a fairness issue — slot recomputation passes for all non-retry bets in the affected epochs. The 5 bets carrying nonce 50 are included in slot recomputation (all pass) and flagged as informational artifacts.
